# BeautyTrack — как устроено приложение

Этот документ объясняет, как работает приложение BeautyTrack «от и до» — без глубокого погружения в код, но с конкретными ссылками на файлы и строки, чтобы было понятно где что находится.

---

## Что такое это приложение вообще

BeautyTrack — это CRM для косметологической клиники. Три типа пользователей: **администратор**, **врач** и **клиент**. Каждый видит своё: врач — только своих клиентов, клиент — только свою карточку, администратор — всё.

Приложение состоит из двух отдельных программ, которые работают вместе:
- **Бэкенд** (`backend/`) — «мозг», хранит данные, отвечает на вопросы
- **Фронтенд** (`frontend/`) — «лицо», то что пользователь видит в браузере

---

## БЭКЕНД

### Технология

Бэкенд написан на **Python** с использованием фреймворка **FastAPI**. FastAPI — это библиотека, которая позволяет очень быстро создавать REST API. REST API — это набор адресов (эндпоинтов), по которым фронтенд обращается за данными.

Данные хранятся в базе данных **SQLite** — это один файл `beautytrack.db` на диске. Для работы с ним используется библиотека **SQLAlchemy**.

### Как организован бэкенд

```
backend/app/
├── main.py        — точка входа, здесь регистрируются все маршруты
├── models.py      — описание таблиц базы данных
├── schemas.py     — описание форматов запросов и ответов
├── config.py      — настройки (секретный ключ, URL базы данных)
├── deps.py        — переиспользуемые зависимости (кто авторизован)
├── security.py    — пароли и JWT-токены
└── routers/       — папка с отдельными файлами для каждой области
    ├── auth.py        — вход/регистрация
    ├── clients.py     — операции с клиентами
    ├── admin.py       — дашборд и управление
    ├── analytics.py   — данные для графиков
    ├── care_plans.py  — планы ухода
    └── ...
```

### База данных: models.py

Файл `backend/app/models.py` — это как чертёж базы данных. Каждый класс Python = одна таблица в базе.

Например, таблица пользователей (строки 48–64):
```python
class User(Base):
    id, email, hashed_password, full_name, role, phone, is_active...
```

Таблица клиентов (строки 87–106):
```python
class Client(Base):
    id, doctor_user_id, user_id, full_name, phone, email, allergies...
```

Связи между таблицами описаны через `relationship` — это позволяет через объект клиента сразу получить его визиты, не пиша дополнительных запросов к базе.

Важно: поле `phone` у Client и User помечено `unique=True` — это значит база данных физически не даст записать двух людей с одинаковым номером.

### Как работают маршруты (роутеры)

Каждый файл в `routers/` отвечает за свою область. Например, `clients.py`:

- `GET /clients` (строка 50) — получить список клиентов
- `GET /clients/me` (строка 42) — получить свою карточку (для роли client)
- `POST /clients` (строка 60) — создать клиента
- `PATCH /clients/{id}` (строка 97) — обновить данные
- `DELETE /clients/{id}` (строка 127) — удалить клиента

Декоратор `@router.get(...)` говорит FastAPI: «когда кто-то стучится по этому адресу, вызови вот эту функцию».

### Авторизация

Используется система **JWT-токенов** (`security.py`). Когда пользователь входит, сервер выдаёт ему «пропуск» — строку из случайных символов (токен). Фронтенд хранит его в `localStorage` браузера и при каждом запросе добавляет в заголовок `Authorization: Bearer <токен>`. Бэкенд проверяет токен и понимает, кто делает запрос.

Проверка ролей сделана через `require_roles()` в `deps.py` — это функция-охранник, которую добавляют к маршрутам. Например, удалить клиента может только врач или администратор.

### Миграции: alembic/

Когда структура базы данных меняется (добавляется поле, убирается таблица), нельзя просто так поменять `models.py` — существующие данные пропадут. Для этого используется **Alembic** — инструмент миграций.

В папке `backend/alembic/versions/` лежат файлы-инструкции:
- `001_bootstrap_schema.py` — создание всех таблиц с нуля
- `002_unique_phone.py` — добавление уникальности телефонов и поля `visit_id` в планы ухода
- `003_link_orphan_clients.py` — связывание «сирот» (клиентов без привязки к аккаунту)

## ФРОНТЕНД

### Технология

Фронтенд написан на **TypeScript** (это JavaScript с указанием типов данных — помогает не делать глупые ошибки) с использованием библиотеки **React 19**. React позволяет строить интерфейс из отдельных кусочков — **компонентов**. Каждая кнопка, карточка, форма — это компонент.

Собирается всё это инструментом **Vite** — он превращает тысячи файлов с кодом в несколько оптимизированных файлов, которые понимает браузер.

### Как организован фронтенд

```
frontend/src/
├── main.tsx           — точка входа, запускает React
├── App.tsx            — корневой компонент, описывает маршруты
├── index.css          — глобальные стили и CSS-переменные
├── auth/
│   └── AuthContext.tsx — хранит информацию о текущем пользователе
├── lib/
│   ├── api.ts         — функция для запросов к бэкенду
│   ├── types.ts       — описание форматов данных
│   ├── theme.ts       — логика тёмной/светлой темы
│   └── ThemeContext.tsx — провайдер темы для всего приложения
├── components/
│   └── ui.tsx         — все переиспользуемые UI-компоненты
└── pages/
    ├── Layout.tsx         — шапка (header) и обёртка всех страниц
    ├── DashboardPage.tsx  — главная страница/дашборд
    ├── ClientsPage.tsx    — список клиентов
    ├── ClientDetailPage.tsx — карточка одного клиента (самый большой файл)
    ├── LoginPage.tsx      — форма входа
    ├── AdminPage.tsx      — панель администратора
    └── dashboard/
        ├── charts/        — четыре виджета с графиками
        └── ...виджеты дашборда
```

### Маршрутизация

В `App.tsx` (строки 40–67) описано, какой компонент показывать по какому адресу браузера:
- `/login` → страница входа
- `/app` → дашборд
- `/app/clients` → список клиентов
- `/app/clients/:id` → карточка конкретного клиента
- `/book/:slug` → публичная онлайн-запись

Это работает через библиотеку **react-router-dom**. Страница не перезагружается при переходе между разделами — это SPA (Single Page Application).

### Запросы к бэкенду: api.ts и React Query

Все обращения к серверу проходят через одну функцию `apiJson()` в файле `lib/api.ts` (строки 36–52). Она:
1. Добавляет токен авторизации к запросу
2. Отправляет запрос на сервер
3. Если сервер ответил ошибкой — бросает её с понятным текстом (функция `formatError` строки 27–34)
4. Если всё ок — возвращает данные в нужном формате

Для кэширования и управления состоянием запросов используется **React Query** (`@tanstack/react-query`). Это библиотека, которая умеет:
- Показывать «Загрузка…» пока данные не пришли (`isLoading`)
- Хранить уже загруженные данные в памяти и не делать лишних запросов
- Обновлять данные после мутации (например, после удаления клиента обновить список)

Пример использования в `ClientDetailPage.tsx` строки 35–48:
```tsx
const clientQ = useQuery({
  queryKey: ['client', clientId],     // уникальный ключ кэша
  queryFn: () => apiJson(`/clients/${clientId}`),  // что загружать
})
```

## КАК СДЕЛАН ИНТЕРФЕЙС (CSS И ФРОНТЕНД-БЛОКИ)

Это самая важная часть — как именно выглядит приложение и откуда взялись все кнопки, карточки и цвета.

### Tailwind CSS — основа стилей

Для стилизации используется **Tailwind CSS v4**. Это очень популярный CSS-фреймворк с необычным подходом: вместо того чтобы писать отдельный CSS-файл, стили применяются прямо в HTML/JSX через специальные классы.

Пример обычного CSS:
```css
.my-button { background: purple; padding: 10px 20px; border-radius: 8px; }
```

Пример с Tailwind:
```jsx
<button className="bg-purple-600 px-4 py-2 rounded-lg">Кнопка</button>
```

Каждое слово в `className` — это один CSS-класс из библиотеки Tailwind. `bg-purple-600` = фиолетовый фон, `px-4` = горизонтальные отступы, `rounded-lg` = скруглённые углы.

**Адаптивность** (разные размеры экрана) делается через префиксы:
- `sm:flex` — применить `flex` только на маленьких экранах и больше
- `lg:grid-cols-2` — двухколоночная сетка только на больших экранах

Пример из `Layout.tsx` строка 38:
```jsx
className="hidden items-center gap-1 rounded-xl bg-zinc-100/80 sm:flex"
```
`hidden` — скрыть, `sm:flex` — показать только начиная с планшета. Так навигация прячется на мобильных.

### Цветовая палитра — кастомные переменные

Фирменный фиолетовый цвет определён в `frontend/src/index.css` строки 4–10:
```css
@theme {
  --color-brand-50: oklch(0.97 0.02 320);   /* очень светлый */
  --color-brand-600: oklch(0.52 0.24 310);  /* основной */
  --color-brand-700: oklch(0.45 0.22 305);  /* тёмный */
}
```

`oklch` — это современный формат цвета (лучше передаёт оттенки чем `#ff0000`). После этого в любом месте приложения можно писать `text-brand-600` или `bg-brand-50` и Tailwind подставит нужный цвет.

### Тёмная тема — CSS-переменные

Тёмная/светлая тема реализована через CSS-переменные в `index.css` строки 14–23:
```css
:root {
  --bg: #fafafa;    /* фон в светлой теме */
  --fg: #18181b;    /* текст в светлой теме */
}
:root.dark {
  --bg: #0b0b0f;    /* фон в тёмной теме */
  --fg: #f4f4f5;    /* текст в тёмной теме */
}
```

Когда пользователь нажимает на кнопку 🌙, JavaScript добавляет класс `dark` к корневому тегу `<html>`. CSS автоматически переключается на тёмные значения переменных.

Логика управления темой живёт в `lib/theme.ts`:
- `getTheme()` — читает сохранённый выбор из `localStorage`
- `applyTheme()` — добавляет/убирает класс `dark` у `<html>`
- `watchSystemTheme()` — следит за настройками ОС

`lib/ThemeContext.tsx` — это «хранилище» темы для всего React-приложения. Любой компонент может узнать текущую тему и изменить её.

### Директива dark: в Tailwind

Для компонентов с тёмными вариантами используется префикс `dark:` прямо в className. В `index.css` строка 3:
```css
@variant dark (&:where(.dark, .dark *));
```
Это говорит Tailwind: «применяй `dark:` классы когда на странице есть класс `.dark`».

Пример из `ui.tsx` строки 51–53 (компонент Card):
```jsx
className="rounded-2xl border border-zinc-200/80 dark:border-zinc-800
           bg-white dark:bg-zinc-900 shadow-sm"
```
`border-zinc-200/80` — граница в светлой теме, `dark:border-zinc-800` — граница в тёмной. Оба класса всегда в разметке, но работает только тот, который нужен сейчас.

## UI-КОМПОНЕНТЫ: что самописное, что из библиотек

### Самописные компоненты (файл `frontend/src/components/ui.tsx`)

Это самый важный файл для понимания визуала. Все базовые элементы интерфейса написаны **с нуля** с использованием только Tailwind-классов. Никакой готовой библиотеки компонентов (типа Material UI или Ant Design) нет — всё самостоятельно.

#### `Card` (строки 45–52) — карточка
Белый прямоугольник со скруглёнными углами и лёгкой тенью. Используется буквально везде: карточки клиентов, блоки на дашборде, формы.
```jsx
<div className="rounded-2xl border border-zinc-200/80 dark:border-zinc-800
                bg-white dark:bg-zinc-900 shadow-sm shadow-zinc-950/5">
```
`rounded-2xl` — сильное скругление, `shadow-sm` — лёгкая тень снизу.

#### `Button` (строки 10–43) — кнопка
Четыре варианта через параметр `variant`:
- `default` — фиолетовая кнопка (основное действие)
- `outline` — белая с рамкой
- `ghost` — прозрачная (для ссылок)
- `destructive` — красная (удаление)

Размеры через `size`: `sm`, `md`, `lg`. Все комбинации описаны в объектах `variants` и `sizes` (строки 21–30). Библиотека `clsx`/`cn` объединяет нужные классы.

#### `Input` (строки 62–72) — поле ввода
Инпут с фокус-эффектом: при фокусе появляется фиолетовая рамка (`focus:border-brand-500 focus:ring-2`). Одинаково выглядит во всех формах.

#### `Tabs` (строки 91–117) — переключатель вкладок
Используется на карточке клиента («Профиль / Визиты / Уход»). Это просто ряд кнопок с разным стилем для активной/неактивной. Сделано **без** библиотеки Radix Tabs — полностью самостоятельно.

#### `ThemeToggle` (строки 119–139) — переключатель темы
Три кнопки ☀️ 💻 🌙 в одном блоке. При нажатии вызывает `setTheme()` из контекста. Активная кнопка подсвечивается белым фоном.

#### `Badge` (строки 78–89) — бейджик роли
Маленький фиолетовый ярлык в шапке («Врач», «Клиент», «Админ»). Фон `bg-brand-50`, текст `text-brand-700`.

### Вспомогательная функция `cn()` (lib/utils.ts)

Функция `cn()` объединяет CSS-классы и разрешает конфликты между ними. Например, если один компонент передаёт `bg-white`, а родитель хочет переопределить на `bg-zinc-100` — `cn()` правильно объединит. Использует библиотеки `clsx` и `tailwind-merge`.

### Что взято из Radix UI

В `package.json` есть несколько пакетов `@radix-ui/*` (dialog, dropdown-menu, tabs, select...). Это библиотека **примитивов** — компонентов без стилей, но с правильным поведением (доступность, клавиатурная навигация). Однако в коде проекта они фактически не используются в текущих страницах — импортов из `@radix-ui` в исходниках нет. Пакеты установлены, но не задействованы.

### Что взято из Recharts (графики)

Графики на дашборде — это **единственное место**, где используется полноценная внешняя библиотека визуализации. Четыре файла в `frontend/src/pages/dashboard/charts/`:

- `Appointments30dWidget.tsx` — линейный график. Из Recharts берётся: `LineChart`, `Line`, `XAxis`, `YAxis`, `CartesianGrid`, `Tooltip`, `ResponsiveContainer`. Мы только указываем данные и цвета (`stroke="var(--color-brand-600)"`).

- `StatusFunnelWidget.tsx` — круговая диаграмма. Из Recharts: `PieChart`, `Pie`, `Cell`, `Legend`, `Tooltip`. Мы задаём цвета сегментов вручную.

- `TopProceduresWidget.tsx` — горизонтальные бары. `BarChart` с `layout="vertical"`.

- `AgeGroupsWidget.tsx` — вертикальные бары. Стандартный `BarChart`.

Каждый виджет обёрнут в наш самописный `<Card>` — так они вписываются в общий дизайн. Данные для графиков приходят с бэкенда через `useQuery` к аналитическим эндпоинтам (`/admin/analytics/...`).

## КАК УСТРОЕНЫ СТРАНИЦЫ

### Шапка и обёртка — Layout.tsx

`Layout.tsx` — это «рамка», которая оборачивает все страницы. Структура (строки 27–78):
```
<div> (весь экран)
  <header>  — шапка с логотипом, навигацией, кнопками
  <main>    — здесь рендерится текущая страница (<Outlet />)
```

Шапка липкая (`sticky top-0`) — остаётся сверху при прокрутке. Фон — белый с размытием (`bg-white/90 backdrop-blur-md`), что даёт красивый «стеклянный» эффект.

Навигационное меню (`<nav>` строки 38–65) скрывается на мобильных (`hidden sm:flex`) — чтобы не занимать место на маленьких экранах.

### Список клиентов — ClientsPage.tsx

Структура страницы:
1. Заголовок + счётчик найденных клиентов
2. Форма «Новый клиент» (только для врача/администратора) — с полями ФИО и Телефон
3. Панель поиска + сортировки
4. Список карточек клиентов

Поиск работает клиентски — функция `useMemo` (строки 55–77) фильтрует уже загруженный массив. Когда пользователь нажимает 🗑 — появляется инлайн-подтверждение (строки 107–128) прямо в карточке, без модального окна.

Важный момент: все `useState` и `useQuery` вызовы находятся **в начале функции** до любых `return` (строки 22–77). Это обязательное требование React — порядок хуков не должен меняться.

### Карточка клиента — ClientDetailPage.tsx

Самый большой файл (1033 строки). Содержит несколько компонентов внутри:

- **`ProfileCard`** — просмотр и редактирование данных клиента. Есть два состояния: режим просмотра и режим редактирования, переключаются кнопкой.
- **`VisitCard`** — карточка одного визита с фотографиями, редактированием, загрузкой фото и формой назначения ухода.
- **`ClientCareView`** — вид для роли клиент: показывает план ухода и рекомендации.
- **`RoutineList`** — утренний/вечерний список средств.
- **`CareHistoryView`** — история планов ухода с привязкой к визитам.

Вкладки «Профиль / Визиты / Уход» реализованы через наш компонент `<Tabs>`.

### Дашборд — DashboardPage.tsx

Для врача/администратора показывает:
- Три плитки с цифрами (записи, отмены, клиенты)
- Сетка виджетов: расписание, быстрые действия, последние визиты, мини-график
- **Четыре аналитических графика** (добавлены в рамках доработок) в сетке 2×2

Для клиента показывает только `<ClientHomeWidget>` — личный виджет с его записями.

| Иконки эмодзи (🗑 ☀️ 🌙) | **Встроенные** в браузер, без библиотек |
| Графики | **Recharts** — библиотека |
| Навигация между страницами | **react-router-dom** — библиотека |
| Загрузка данных / кэш | **React Query (@tanstack/react-query)** — библиотека |
| Форматирование дат | **date-fns** — библиотека |
| Объединение CSS-классов | **clsx + tailwind-merge** (функция `cn()`) — библиотеки |
| @radix-ui пакеты | Установлены, но **не используются** в текущих страницах |
| Шапка, страницы, формы, логика | **Самописные** — папка `pages/` |
| Аутентификация (контекст) | **Самописная** — `AuthContext.tsx` |
| REST API (бэкенд) | **FastAPI** — библиотека-фреймворк |
| База данных ORM | **SQLAlchemy** — библиотека |
| Миграции БД | **Alembic** — библиотека |
| Пароли + JWT | **bcrypt + python-jose** — библиотеки |
